# Legacy Tracker Snapshot

## Status

Archived working tracker from the earlier exploratory phase of the Solomon MVP evaluation-spec effort.

This file is preserved for historical traceability only.

It should **not** be treated as the current source of truth.

Current source-of-truth documents are:

- `docs/01_foundations_and_architecture.md`
- `docs/02_Operations and Evaluator Workflow`

---

## Purpose of the tracker during drafting

This tracker was used to:

- sequence the work
- keep open questions visible
- capture design decisions as they emerged
- identify next artifacts to draft
- track the evolution from conceptual discussion to developer-facing specification

It functioned as a working notebook rather than a final deliverable.

---

The tracker captured work across these themes:

- definition of Solomon as an expert-in-the-middle mediation platform
- evaluation-first architecture strategy
- synthetic case generation strategy
- core vs plugin boundary
- divorce as the first plugin
- core mediation competency model
- weighting of competency families
- escalation taxonomy and threshold logic
- benchmark scenario design
- evaluator score sheet
- evaluator instructions
- divorce template families
- repository packaging strategy

---

## Why this file is archived

This tracker is archived because its contents were later consolidated into cleaner documents.

Compared with the active docs, the tracker is:

- more iterative
- more repetitive
- less stable in terminology
- less convenient for developer onboarding

It remains useful only for:
- project history
- rationale archaeology
- tracing how certain design choices evolved

---

## Archive guidance

If referenced at all, this file should be used only to answer questions such as:

- how did a design decision emerge?
- what alternatives were considered?
- what sequencing logic led to the current package?

It should not be used to define current implementation behavior.
