# Post-Practical Trial Memo 027

## Purpose

This memo records the findings from the first practical trial / validation pass over the complete divorce corpus using [PRACTICAL-TRIAL-PROTOCOL-001-complete-divorce-corpus-v0.md](./PRACTICAL-TRIAL-PROTOCOL-001-complete-divorce-corpus-v0.md).

The aim of the trial was not to add breadth. It was to determine whether the completed divorce corpus is practically reviewable, evaluator-usable, and trustworthy enough to guide the next phase.

---

## Short Result

The practical trial was successful.

The divorce corpus now looks:

- family-complete
- evaluator-usable
- support-artifact-usable
- practically reviewable without heavy transcript dependence

The trial did **not** reveal a strong immediate need for a second plugin family.

The current evidence points instead toward a lighter next phase:

- selective divorce-trial remediation where the practical pass felt comparatively thin
- continued evaluator/artifact refinement only where it materially improves practical review

---

## Pass A Findings: Bounded / Caution Calibration

Reviewed:

- `D-B04`
- `D-B05`
- `D-B06`
- `D-B07`
- `D-B10`
- `D-B11`

### What held up well

- `D-B04` still functions as the bespoke caution anchor rather than collapsing into the patterned slices.
- `D-B05`, `D-B06`, and `D-B07` remain clearly distinguishable package slices with different domain emphasis.
- `D-B10` still reads as emotionally intense but workable, which is a crucial contrast against `D-B12`.
- `D-B11` still preserves information/confidence asymmetry as a caution problem rather than a hidden severe-escalation case.

### Practical-review judgment

This cluster is practically reviewable and does not show strong posture drift.

The trial did not suggest that these bounded/caution slices are over-escalating or artifact-thin in a way that blocks practical use.

### Remaining light weakness

The lighter `M0` package slices still have less committed evaluator/support depth than the headline higher-mode slices. That is acceptable for now, but they remain more dependent on summary and review-path framing than the strongest anchored cases.

## Pass B Findings: Higher-Mode Co-Handling Calibration

Reviewed:

- `D-B08`
- `D-B12`

### What held up well

- `D-B08` remains clearly a fairness/process-breakdown case.
- `D-B12` remains clearly an emotional-flooding / failed-repair case.
- The two slices are now distinguishable at the evaluator-summary level without needing transcript-heavy interpretation.
- The escalation logic in both slices reads as “repair first, co-handle when no longer stabilizing,” but for meaningfully different reasons.

### Practical-review judgment

This cluster passed the practical trial well.

The distinction between:

- domination / participation imbalance
- emotional flooding / failed stabilization

is now clear enough to support practical review and calibration.

### Remaining light weakness

`D-B12` is now evaluator-anchored, but still has a thinner committed support-artifact layer than `D-B08`. That is no longer a blocking asymmetry, but it is still the clearest place where future refinement could improve reviewer ease.

---

## Pass C Findings: Hard-Trigger / Complexity Contrast

Reviewed:

- `D-B13`
- `D-B14`
- `D-B09`

### What held up well

- `D-B13` remains clearly about constrained voluntariness / safety-compromised participation.
- `D-B14` remains clearly about impaired participation capacity rather than coercion.
- `D-B09` still reads as complexity-beyond-safe-autonomy rather than a disguised hard-trigger case.
- The three-slice contrast is now good enough that practical review can distinguish:
  - unsafe participation due to pressure
  - unreliable participation due to capacity impairment
  - legitimate stop-for-review due to complexity

### Practical-review judgment

This cluster is strong and is one of the clearest signs that the divorce corpus is now more than a collection of slices. It behaves like a calibrated family.

### Remaining light weakness

`D-B09` still has lighter evaluator/support anchoring than the two hard-trigger slices. That does not undermine the family distinction, but it is the clearest optional deepening candidate if evaluator usability becomes the next priority.

---

## Artifact And Evaluator Usability Findings

### Strengths

- Higher-mode summaries now preserve reason-for-posture clearly.
- Reference evaluator packages are strong enough to support comparative review.
- Support artifacts are now genuinely useful on the slices where they are committed, rather than merely decorative.
- The review paths are good enough that an evaluator can understand what to compare and why.

### Weaknesses

- Support-artifact depth is still uneven across the corpus.
- Some slices are strongly anchored by full reference sets while others rely mainly on evaluation summaries and review-path prose.
- The corpus is usable now, but not yet fully symmetric in how “review-ready” each slice feels.

---

## Boundary Findings

The practical trial did **not** surface a strong unresolved boundary problem that requires a second plugin immediately.

This is important.

The current core/plugin/benchmark boundary appears good enough to support a practical divorce-first phase.

Possible future reasons for a second plugin still exist, but they are not current blockers. Nothing in this trial suggested:

- shared artifact language is still fatally divorce-shaped
- evaluator helpers are unusably single-plugin in practice
- or the plugin interface cannot be trusted until another domain exists

So a second plugin remains a later strategic move, not a trial-driven necessity.

---

## Recommendation

The next phase should **not** be immediate second-plugin development.

The best next move is:

1. do a small targeted remediation / refinement pass based on practical-trial usability, especially where evaluator/support anchoring still feels uneven
2. then reassess whether the next work should be:
   - more evaluator/artifact refinement
   - or preparation for a second plugin family

---

## Most Likely Near-Term Priorities

If we continue with the divorce corpus first, the best candidates are:

- deepen `D-B12` support-artifact anchoring
- optionally deepen `D-B09` evaluator/support anchoring
- smooth any remaining review-path asymmetries across the corpus

These are now refinement choices, not structural rescue work.

---

## Bottom Line

The divorce plugin has now crossed an important threshold:

it is no longer just internally well-built; it is practically trialable.

That means the project can stay disciplined:

- trial divorce first
- refine where the trial actually pointed
- only then decide whether a second plugin is needed for perspective
