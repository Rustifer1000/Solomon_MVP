# Solomon Project Overview

Solomon is a mediation AI project aimed at building a universal core mediation engine: a system that can support structured, high-quality mediation across many domains without collapsing those domains into one generic workflow. The long-term vision is a reusable core that handles the fundamental mediation functions well: framing, issue clarification, interest elicitation, option exploration, bounded process guidance, uncertainty handling, and appropriate escalation. Around that core, domain-specific plugins can add the specialized logic needed for settings like divorce mediation, workplace conflict, business disputes, or other structured negotiation environments.

The key idea is that Solomon should not be a judge, arbitrator, or autonomous decision-maker. Its purpose is not to impose outcomes, but to help parties and mediators work through conflict more clearly, safely, and productively. Human involvement is not treated as a failure mode; in many cases it is the right and intended success mode. That means the system must preserve uncertainty honestly, distinguish facts from positions and missing information, and know when to continue cautiously rather than overclaiming confidence.

This project is being developed evaluation-first. Before trying to build a production mediation product, the goal is to create a trustworthy executable evaluation system that can test whether the underlying mediation architecture is actually sound. That evaluation system is meant to run benchmark cases, produce reviewable artifacts, and allow evaluators to judge whether Solomon is behaving correctly from emitted evidence rather than from vague impressions. In that sense, the current executable system is not the final mediation product; it is the engine for testing whether the future product would deserve trust.

Architecturally, Solomon is organized around several distinct layers. The core mediation layer handles general mediation process logic. Domain plugins contribute specialized qualification and caution logic for particular case types. The platform/runtime layer manages state, normalization, escalation, and artifact generation. The evaluator layer remains separate so that success can be judged from outputs rather than assumed by the runtime itself. This separation matters because one of the project's main goals is to prevent fluent language from masquerading as trustworthy mediation reasoning.

Where we are right now: developing the first executable MVP evaluation runtime for that vision. The repo has moved beyond idea and prose specification into an early working prototype. It can run a benchmark slice, normalize turn outputs into a typed contract, mutate authoritative state, apply plugin assessment and escalation logic, and emit evaluator-facing artifacts. The immediate task is not yet to build the universal mediation product itself, but to make this evaluation runtime trustworthy enough to tell us whether the universal core mediation engine is actually being built in the right way.
